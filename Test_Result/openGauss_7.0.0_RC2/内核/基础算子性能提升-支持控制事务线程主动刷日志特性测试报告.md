# 1 概述

openGauss在低并发基础算子场景xlog落盘明显落后于竞品，本特性主要针对openGauss低并发基础算子场景的xlog落盘的业务流程进行优化。为了提升低并发基础算子性能，本特性添加了guc参数可以修改xlog落盘逻辑，针对性提升了一些基础算子用例的性能。针对该需求出具本测试报告。

# 3 版本概要测试结论、关键风险和规避措施

## 3.1 测试结论总结

需求共计执行8个用例，主要覆盖了参数功能测试、性能测试、可靠性测试等场景，共发现4个问题单，整体质量一般。

| 测试活动                                                     | 活动评价                                                     |
| ------------------------------------------------------------ | ------------------------------------------------------------ |
| xlog落盘逻辑修改用户不感知，需要测试特性对xlog落盘逻辑的修改是否影响正常功能。需要在备份恢复后校验xlog文件是否正确，数据是否和原来一致，主备数据是否一致 | 备份恢复后校验xlog文件正确，数据和原来一致，主备数据一致     |
| 参数enable_LC_xlog_flush_optimize功能测试                    | enable_LC_xlog_flush_optimize的默认值为off，使用三种修改方式可以修改参数enable_LC_xlog_flush_optimize的值，修改过后的值可以正常生效，数据库升级后可以修改对应的参数值，符合预期。 |
| 基础算子测试——针对以下用例进行性能测试，单机场景预计性能提升15%，传统主备和资源池化性能不下降<br/>sigle_pbe_delete_by_id_btree.sql<br/>sigle_pbe_delete_by_id_multi.sql<br/>sigle_pbe_delete_by_id_primary.sql<br/>sigle_pbe_delete_by_idx_multi.sql<br/>sigle_pbe_insert_btree.sql<br/>sigle_pbe_insert_multi.sql<br/>sigle_pbe_insert_noindex.sql<br/>sigle_pbe_insert_primary.sql<br/>sigle_pbe_update_id_by_id_btree.sql<br/>sigle_pbe_update_id_by_id_primary.sql<br/>sigle_pbe_update_idx_by_id_btree.sql<br/>sigle_pbe_update_idx_by_id_primary.sql | 使用指定调优参数对以下用例进行性能测试：<br/>极简版用例在引入特性前耗时平均值为340.9ms，引入特性后耗时平均值为241.1ms，各用例平均性能提升为41.67%<br/>企业版用例在引入特性前耗时平均值为347.09ms，引入特性后耗时平均值为247.75ms，各用例平均性能提升为40.39%<br/>轻量版用例在引入特性前耗时平均值为2139.61ms，引入特性后耗时平均值为317.82ms，各用例平均性能提升为578.04%<br/>传统主备和资源池化场景在给定参数下用例执行时间过长，但各sql语句在单条执行时，引入特性前后性能几乎不变，以上结果符合预期 |
| 性能测试时需要后台调用iostat命令观察记录IO是否出现明显增大   | IO确实出现明显增大，引入特性前，测试过程中磁盘占用率的峰值为整个磁盘的12%左右，引入特性后，测试过程中磁盘占用率的峰值可达整个磁盘的30%以上，确实出现明显增大 |
| 性能测试过程中调用检查内存命令观察是否出现内存泄漏           | 未出现内存泄漏                                               |
| 资源池化场景添加实时构建等特性查看功能是否使用正常           | 功能使用正常                                                 |
| 传统主备和资源池化制造故障，观察重启后是否功能正常，xlog方面是否出现影响 | 对传统主备和资源池化制造kill -9和kill -19故障，重新建立主备关系之后，主备间数据一致，重建前的部分没有出现数据丢失，xlog写入可以正常进行，主备切换备机升主正常，业务可以正常进行 |
| 导入数据前后记录xlog位点，并使用PITR恢复                     | PITR可以按时间点进行恢复，恢复后的数据和插入的数据一致，不会出现漏xlog等其他问题 |
